# 全能格式转换器（Transcoder）设计文档（阶段1）

## 结论摘要

- 采用“**解复用 -> 解码 -> 重编码 -> 复用**”的标准转码流水线，最稳妥、最容易教学和扩展。
- 默认只保留 **1 路视频 + 1 路音频**，主动忽略字幕流，避免 MP4 兼容性和初学者复杂度问题。
- 以 `video: libx264 + CRF + preset`、`audio: aac` 作为默认方案，兼顾兼容性、质量与速度。
- 音画同步核心是“**正确维护时间基与时间戳重映射**”：所有跨模块时间戳都必须 `rescale` 到目标 `time_base`。
- 验收不靠“主观看起来正常”，而是结合 `ffprobe` 自动化检查时长差、码率、流类型与播放器兼容性。

---

## 1. 需求拆解

### 1.1 功能需求

1. 输入任意常见封装格式（如 MKV、MOV、TS、AVI）。
2. 输出统一为 MP4。
3. 流选择规则：只输出视频流与音频流，忽略字幕流。
4. 可配置编码参数：
   - 视频：`CRF`、`preset`
   - 音频：码率（如 `128k`、`192k`）
5. 输出音画同步。

### 1.2 非功能需求

- **可用性**：命令行参数清晰，错误提示可读。
- **可维护性**：模块化设计，便于后续扩展（多音轨、字幕、硬编）。
- **兼容性**：生成 MP4 能在常见播放器播放（VLC、QuickTime、Chrome）。
- **可观测性**：输出关键日志（输入信息、选择的流、编码参数、进度、异常）。

### 1.3 边界条件

- 仅音频输入：允许输出“仅音频 MP4”（或根据策略直接报错，建议可配置）。
- 仅视频输入：允许输出“无音轨 MP4”。
- 多音轨：默认选第一路，可通过参数指定语言或索引。
- 无可用视频且无可用音频：报错退出。
- 输入损坏：尽量容错读取，严重损坏则失败并返回明确错误码。

## 2. 总体架构设计

### 2.1 模块划分

1. **CLI 参数解析模块**
   - 解析输入输出路径、CRF、preset、音频码率、流选择策略。
2. **输入探测模块（Probe）**
   - 打开媒体，读取 stream 信息（codec、time_base、duration）。
3. **流选择模块（Stream Selector）**
   - 确定 1 路视频和 1 路音频，过滤字幕流。
4. **解复用模块（Demux）**
   - 从容器读取 `AVPacket`（压缩数据包）。
5. **解码模块（Decode）**
   - 将压缩 `packet` 解为原始 `frame`（像素/PCM）。
6. **编码模块（Encode）**
   - 将 `frame` 编为目标编码（H.264/AAC）。
7. **复用模块（Mux）**
   - 将编码后的 packet 写入 MP4，处理全局 header、尾部 trailer。
8. **时间戳同步模块（Timestamp Manager）**
   - 统一处理 `PTS/DTS` 与 `time_base` 转换。
9. **日志与错误处理模块**
   - 错误码分层、关键路径日志、异常兜底释放资源。

### 2.2 数据流/时序流转（Mermaid）

```mermaid
flowchart LR
    A[输入文件] --> B[Probe读取流信息]
    B --> C[选择视频流和音频流]
    C --> D[Demux读取Packet]
    D --> E[Decode得到Frame]
    E --> F[Encode得到新Packet]
    F --> G[时间戳重映射rescale]
    G --> H[Mux写入MP4]
    H --> I[输出文件]
```

为什么这样设计：
- 这是 FFmpeg 最标准路径，资料多、社区经验多，初学者最容易定位问题。
- 每个模块职责单一，后续加硬件编码或滤镜时不需要推翻整体结构。

## 3. 核心原理讲解（初学者友好）

### 3.1 容器 vs 编码格式

- **容器（Container）**：像“盒子”，例如 MP4、MKV，负责把音视频流装在一起。
- **编码格式（Codec）**：像“压缩算法”，例如 H.264、AAC，决定数据怎么压缩。

白话：`MKV` 和 `MP4` 是包装袋；`H.264`、`AAC` 是袋子里的内容压缩方式。

### 3.2 Stream / Packet / Frame

- **Stream**：媒体中的一条轨道（视频轨、音频轨、字幕轨）。
- **Packet**：压缩后的数据块（还不能直接渲染）。
- **Frame**：解码后的原始数据（视频帧像素、音频采样）。

白话：`packet` 是“压缩包”，`frame` 是“解压后的可用数据”。

### 3.3 time_base / PTS / DTS / duration

- **time_base**：时间单位，比如 `1/90000`，表示“1 tick = 1/90000 秒”。
- **PTS（Presentation Timestamp）**：显示/播放时间。
- **DTS（Decoding Timestamp）**：解码顺序时间。
- **duration**：持续时长，单位也是当前 `time_base`。

常见换算：
- 秒 = `pts * av_q2d(time_base)`
- 不同 time_base 转换：`av_rescale_q(value, src_tb, dst_tb)`

### 3.4 为什么会音画不同步

常见成因：
1. 直接复制时间戳，不做 `time_base` 转换。
2. 编码器重排（尤其 B 帧）导致 DTS/PTS 处理错误。
3. 音频采样率变更后，frame 数与时间戳计算不一致。
4. 封装前未保证时间戳单调递增（mux 失败或播放异常）。

修复思路：
- 所有跨上下文时间戳都走 `rescale`。
- 使用 FFmpeg 提供的编码/复用 API 让库完成一部分重排。
- 统一在写封装前做时间戳校正与单调性检查。

### 3.5 核心参数与概念补充（Q&A）

#### 1. sample_aspect_ratio (SAR) 是什么？
- **概念**：单个像素的物理宽高比（Pixel Aspect Ratio）。
- **作用**：现代显示器像素通常是正方形（1:1），但很多传统视频标准（如 DVD NTSC/PAL）像素是长方形的。
- **与 DAR 的关系**：显示宽高比 (DAR) = (视频宽度 / 视频高度) × 像素宽高比 (SAR)。
- **读写规则**：编码时由用户设置（通常设为 `{1, 1}`），解码时由 `libavcodec` 自动从视频码流中读取。

#### 2. gop_size 是必填的吗？
- **回答**：不是严格必填，但**强烈建议填写**。如果不填，会使用底层编码器（如 x264）的默认值（通常是 250）。
- **为什么重要**：
  - **影响 Seek（拖动进度条）体验**：播放器只能从 I 帧开始解码。GOP 太大（如 10 秒一个 I 帧）会导致拖动进度条时卡顿或定位不准。
  - **影响流媒体切片**：HLS/DASH 切片必须以 I 帧开头，GOP 太大会影响切片粒度和秒开率。
- **推荐设置**：通常设置为**视频帧率（fps）的 1 倍到 2 倍**（例如 25fps 的视频，GOP 设为 25 或 50）。

#### 3. CRF（Constant Rate Factor）是动态调整的吗？
- **回答**：**设置层面是固定的，执行层面是动态的。**
- **原理**：你对整个视频只设置一个固定的 CRF 值（例如 23），代表你期望的“恒定视觉质量”。但在逐帧编码时，编码器会根据画面复杂度**动态调整码率**。
- **效果**：对于运动剧烈的画面，人眼对细节不敏感，编码器分配较少码率（高压缩）；对于静态画面，人眼对瑕疵敏感，编码器分配极高码率（低压缩）。这就像变频空调，设定温度固定，但输出功率动态变化。

#### 4. AVFMT_GLOBALHEADER 标志位的作用是什么？
- **背景**：视频（SPS/PPS）和音频（AudioSpecificConfig）的配置信息存放位置因容器而异。
  - **带内传输（如 .ts）**：配置信息打包在每一个关键帧前面。
  - **带外传输（如 .mp4, .flv）**：配置信息统一提取放在文件头部（如 MP4 的 `moov` box）。
- **代码逻辑**：`if (ofmt_ctx->oformat->flags & AVFMT_GLOBALHEADER)` 用于判断当前输出容器是否要求配置信息放在头部。如果是，则给编码器打上 `AV_CODEC_FLAG_GLOBAL_HEADER` 标记，让编码器生成全局头信息，避免 MP4 播放兼容性问题。

## 4. 转码策略设计

### 4.1 视频策略（H.264）

- 编码器：`libx264`
- 质量控制：`CRF`（推荐默认 `23`）
- 速度控制：`preset`（推荐默认 `medium`）

CRF 结论（越小越清晰，也越大）：
- 高质量：`18~21`
- 平衡：`22~24`
- 高速省算力：`25~28`

preset 结论（越慢压缩率越高）：
- `veryfast`：速度快，体积偏大
- `medium`：平衡
- `slow`：更省体积但更耗时

### 4.2 音频策略（AAC）

- 编码器：`aac`
- 默认码率：`128k`（语音/普通内容够用）
- 音乐或高保真需求：`192k~256k`

### 4.3 MP4 封装兼容性

- 使用 `-movflags +faststart`：把元数据前置，网页端可更快起播。
- 像素格式建议输出 `yuv420p`，兼容性最好。

关键命令示例（CLI 版本）：

```bash
ffmpeg -i input.mkv \
  -map 0:v:0 -map 0:a:0 -sn \
  -c:v libx264 -crf 23 -preset medium -pix_fmt yuv420p \
  -c:a aac -b:a 128k \
  -movflags +faststart \
  output.mp4
```

参数解释（白话）：
- `-map 0:v:0 -map 0:a:0`：选第一路视频和第一路音频。
- `-sn`：禁用字幕输出。
- `-crf/-preset`：控制画质、速度、体积平衡。
- `-movflags +faststart`：优化网络播放首开速度。

## 5. 流选择策略

### 5.1 默认规则

1. 视频：选择第一路“可解码视频流”。
2. 音频：选择第一路“可解码音频流”。
3. 字幕：全部忽略。

### 5.2 可配置扩展

- `--video-index` 指定视频流索引。
- `--audio-index` 指定音频流索引。
- 后续扩展 `--audio-lang zh` 按语言选轨。

为什么先忽略字幕：
- MVP 目标先保证主路径稳定（音视频转码 + 同步）。
- 字幕涉及文本/位图字幕、时轴转换、封装兼容，复杂度高，适合二期扩展。

## 6. 同步与时间戳设计

## 6.1 统一时钟/主时钟思路

- 离线转码场景下，不是实时播放，不需要“墙钟”。
- 采用“各流各自时轴 + 复用器规则”方案：
  - 视频按视频编码输出时间戳推进
  - 音频按采样数推进
  - 写入前统一转换到输出流 `time_base`

### 6.2 时间戳重映射与 rescale 原理

原则：**谁生产时间戳，就用谁的 time_base；写给谁，就转到谁的 time_base。**

典型步骤：
1. 解码前：输入 `packet.pts/dts` 在输入流 `in_tb`。
2. 编码后：输出 `packet.pts/dts` 在编码器 `enc_tb`。
3. 复用前：`packet` 需转换到输出流 `out_tb`。

伪代码：

```text
pkt->pts = av_rescale_q_rnd(pkt->pts, enc_tb, out_tb, ...);
pkt->dts = av_rescale_q_rnd(pkt->dts, enc_tb, out_tb, ...);
pkt->duration = av_rescale_q(pkt->duration, enc_tb, out_tb);
```

### 6.3 丢帧/补帧/重采样策略

- 视频：
  - 若输入 VFR（可变帧率），默认保持时间戳驱动，不强行 CFR。
  - 仅在用户指定固定帧率时才做重复/丢帧策略。
- 音频：
  - 采样率不一致时走重采样（`swr`）。
  - 按采样点数计算每帧持续时长，确保音频时轴连续。

### 6.4 常见坑与规避

1. **坑**：把 `in_tb` 的 pts 直接写到 `out_tb`  
   **规避**：每次写封装前都 `av_packet_rescale_ts` 或等价处理。
2. **坑**：编码 flush 阶段漏包  
   **规避**：`send_frame(NULL)` 后持续 `receive_packet` 到 `EAGAIN/EOF`。
3. **坑**：只看 PTS 不看 DTS 导致 mux 报 non-monotonic  
   **规避**：确保写入前 DTS 单调。

## 7. 错误处理与健壮性

### 7.1 错误码规范（建议）

- `0`：成功
- `10x`：参数错误（输入路径、参数范围）
- `20x`：输入打开/探测失败
- `30x`：解码失败
- `40x`：编码失败
- `50x`：封装写入失败

### 7.2 日志分级

- `INFO`：流程节点（打开输入、选择流、开始转码）
- `WARN`：可恢复异常（个别坏包跳过）
- `ERROR`：不可恢复错误（编码器初始化失败）

### 7.3 失败策略

- 初始化失败立即退出，避免产生半成品。
- 转码中遇到单个坏包可配置“跳过并告警”。
- 始终执行资源释放（context、frame、packet、file handle）。

## 8. 性能与质量权衡

### 8.1 三角关系

- 速度快 -> 压缩率一般 -> 文件更大
- 画质高 -> 码率更高或编码更慢
- 体积小 -> 可能牺牲画质或耗时增加

### 8.2 参数模板（可直接用）

1. 高质量（慢）  
   `-crf 20 -preset slow -b:a 192k`
2. 平衡（默认）  
   `-crf 23 -preset medium -b:a 128k`
3. 高速（体积偏大）  
   `-crf 26 -preset veryfast -b:a 128k`

## 9. 测试与验收方案

### 9.1 功能测试

- 输入 MKV（含视频+音频+字幕） -> 输出 MP4 仅保留视频+音频。
- 输入仅音频、仅视频、多音轨文件，验证边界行为。

### 9.2 同步测试

- 播放器主观试听：是否出现明显口型漂移。
- `ffprobe` 客观检查：
  - 视频时长 vs 音频时长差值（阈值如 <= 80ms）
  - 是否存在异常起始时间偏移

示例命令：

```bash
ffprobe -v error -show_streams -show_format -print_format json output.mp4
```

关注字段：
- `streams[].codec_type`
- `streams[].duration`
- `format.duration`
- `streams[].start_time`

### 9.3 兼容性测试

- VLC / QuickTime / 浏览器（video 标签）可播放
- 拖动进度条正常
- 首帧加载速度符合预期（`faststart` 生效）
