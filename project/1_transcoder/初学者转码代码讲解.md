# `main.cpp` 初学者转码代码讲解

这份文档结合 `src/main.cpp` 来讲解这份转码器代码，目标不是“背 API”，而是帮你把下面这件事真正串起来：

**输入文件是怎么一步一步变成输出文件的。**

---

## 第一部分：先看整体图景

### 1. 什么叫转码？

最通俗的理解：

- 输入文件里本来装的是一份“压缩过的音视频数据”
- 程序先把它**解压**成原始数据
- 再按新的要求**重新压缩**
- 最后再装进新的输出文件

比如：

- 输入：`input.mkv`
- 视频编码：H.265
- 音频编码：Opus
- 输出：`output.mp4`
- 视频编码改成：H.264
- 音频编码改成：AAC

这就叫**转码（Transcode）**。

### 2. 它和 remux 有什么区别？

这是 FFmpeg 初学者最容易搞混的点之一。

#### remux（重封装 / 直拷贝）

只换“盒子”，不动里面的数据。

例如：

- 原来：`MKV` 容器 + `H.264/AAC`
- 目标：`MP4` 容器 + `H.264/AAC`

因为编码格式没变，所以只需要：

- 读输入 `AVPacket`
- 改时间戳刻度
- 写进输出文件

不需要：

- 解码
- 编码
- 像素格式转换
- 音频重采样

这份代码里的 `copy_mode_run()` 就是在做这个。

#### transcode（转码）

不仅换“盒子”，还把里面的数据重新压缩。

例如：

- 原来视频是 H.265，要改成 H.264
- 原来音频是别的格式，要改成 AAC
- 原来像素格式不适合编码器，要转成 `yuv420p`
- 原来音频采样格式不适合编码器，要重采样/重排布

这时就必须经过：

`decode -> convert -> encode`

这份代码里的 `transcode_mode_run()` 就是在做这个。

### 3. 整个程序的大流程

可以先把这份代码理解成下面这个流程：

```text
输入文件
-> 打开输入容器
-> 读取流信息
-> 选择要处理的视频流/音频流
-> 创建输出容器
-> 为每一路流准备 StreamContext
-> 打开解码器
-> 打开编码器
-> 写输出文件头
-> 主循环读取输入 AVPacket
-> 解码成 AVFrame
-> 视频/音频格式转换
-> 重新编码成新 AVPacket
-> 写入输出容器
-> flush 解码器和编码器
-> 写文件尾
-> 释放资源
```

如果是 copy 模式，则中间会简化成：

```text
输入文件
-> 打开输入容器
-> 读取流信息
-> 创建输出容器
-> 建立输入流索引 -> 输出流索引映射
-> 读输入 AVPacket
-> 时间戳换算
-> 直接写输出容器
-> 写文件尾
```

## 第二部分：最重要的几个概念

### 1. `AVFormatContext`

它是什么：

- 它代表一个“媒体容器上下文”
- 你可以把它理解成“整个媒体文件的总管理对象”

在这份代码里：

- `ifmt_ctx` 代表输入文件
- `ofmt_ctx` 代表输出文件

它负责的事：

- 记录容器里的所有流
- 管理文件读写
- 写 header / trailer
- 决定用哪个 muxer / demuxer

它和其他对象的关系：

- `AVFormatContext` 下面有很多个 `AVStream`
- `AVStream` 表示容器里的每一路流

可以理解成：

- `AVFormatContext` 是“整栋楼”
- `AVStream` 是“楼里的每个房间”

### 2. `AVStream`

它是什么：

- 表示输入或输出容器中的“一路流”
- 比如视频流、音频流、字幕流

在这份代码里，它主要用来：

- 识别当前流是不是视频/音频
- 读取输入流的 `codecpar`
- 获取输入流 `time_base`
- 创建输出流时保存输出参数

它和其他对象的关系：

- `AVFormatContext` 管很多个 `AVStream`
- `AVStream` 本身不是解码器
- 它更像“流的说明书”

### 3. `AVCodecContext`

它是什么：

- 编解码器真正工作的上下文
- 你可以把它理解成“干活的机器”

在这份代码里：

- `dec_ctx`：负责解码
- `enc_ctx`：负责编码

它负责的事：

- 解码器：把压缩包 `AVPacket` 解成原始帧 `AVFrame`
- 编码器：把原始帧 `AVFrame` 编成压缩包 `AVPacket`

和 `AVStream` 的区别：

- `AVStream` 更像流的描述信息
- `AVCodecContext` 才是真正参与编解码工作的对象

### 4. `AVPacket`

它是什么：

- 压缩后的数据块
- 来自输入容器，或者来自编码器输出

可以理解成：

- 这是“压缩包”
- 还不能直接拿去显示或播放原始内容

在这份代码里：

- `av_read_frame()` 读出来的是它
- `avcodec_send_packet()` 送给解码器的是它
- `avcodec_receive_packet()` 从编码器拿出来的也是它
- 最后写进输出容器的还是它

### 5. `AVFrame`

它是什么：

- 解码后的原始数据
- 对视频来说是原始像素帧
- 对音频来说是原始 PCM 样本

可以理解成：

- `AVPacket` 是压缩包
- `AVFrame` 是解压后的可直接处理数据

在这份代码里：

- 解码器输出它
- 视频用它做 `sws_scale`
- 音频用它做 `swr_convert`
- 编码器输入也是它

### 6. `Options`

它是什么：

- 程序自己的参数结构体
- 用来保存命令行参数解析结果

在这份代码里，它保存：

- 输入路径
- 输出路径
- CRF
- preset
- 音频码率
- 是否 copy 模式
- 指定视频流/音频流索引
- 按语言选择音轨

### 7. `StreamContext`

这是这份代码最关键的自定义结构体。

它是什么：

- 程序自己定义的“每一路流的总控对象”

它不是 FFmpeg 自带的，而是代码作者为了管理方便自己设计的。

它保存了这一路流从输入到输出需要的所有关键状态：

- 这一路是音频还是视频
- 输入流索引
- 输出流索引
- 解码器 `dec_ctx`
- 编码器 `enc_ctx`
- 视频转换器 `sws_ctx`
- 音频重采样器 `swr_ctx`
- 音频 FIFO
- 输出时间戳游标

为什么需要它：

如果没有它，代码里就会充满：

- `video_dec_ctx`
- `audio_dec_ctx`
- `video_enc_ctx`
- `audio_enc_ctx`
- `video_sws_ctx`
- `audio_swr_ctx`
- `video_output_index`
- `audio_output_index`

代码很快会失控。

而有了 `StreamContext` 以后，主循环只需要：

- 找到这个输入包属于哪个 `StreamContext`
- 然后按照 `sc.type` 继续处理

### 8. `AVPacket` 和 `AVFrame` 的区别

#### `AVPacket`

- 压缩数据
- 更接近“文件里的内容”
- 输入容器读出来的是它
- 编码器输出的也是它

#### `AVFrame`

- 原始数据
- 更接近“真正的画面和声音”
- 解码器输出的是它
- 编码器输入的是它

一句话：

**Packet 是压缩态，Frame 是原始态。**

### 9. `AVStream` 和 `StreamContext` 的区别

#### `AVStream`

- FFmpeg 自带
- 表示容器中的一路流
- 更像“流描述”

#### `StreamContext`

- 代码自己定义
- 表示程序对一路流的完整处理状态
- 更像“这一路流在程序里的总控对象”

一句话：

- `AVStream` 是容器里的“流说明书”
- `StreamContext` 是程序里的“处理工作台”

## 第三部分：按执行顺序一步一步带你走

### 第 1 步：解析参数

代码入口：

- `parse_args(argc, argv, opt)`

输入：

- 命令行参数

输出：

- 填充好的 `Options opt`

为什么必须有：

- 程序必须知道输入输出路径、编码参数、是否 copy 等

### 第 2 步：打开输入文件

代码：

```cpp
ret = avformat_open_input(&ifmt_ctx, opt.input_path.c_str(), nullptr, nullptr);
```

输入：

- 输入文件路径

输出：

- `ifmt_ctx`

为什么必须有：

- 不打开文件，后面无法读流信息和媒体包

### 第 3 步：探测流信息

代码：

```cpp
ret = avformat_find_stream_info(ifmt_ctx, nullptr);
```

输入：

- 已打开的输入上下文

输出：

- 流信息更完整的 `ifmt_ctx`

为什么必须有：

- 分辨率、编码格式、时间基等很多信息要靠这一步探测

### 第 4 步：选择视频流和音频流

代码：

- `select_first_stream(...)`
- `select_audio_stream(...)`

输入：

- 输入容器里的所有流

输出：

- `video_index`
- `audio_index`

为什么必须有：

- 一个文件可能有多轨音频、多轨视频、字幕流
- 程序要明确“到底处理哪一路”

### 第 5 步：创建输出上下文

代码：

```cpp
ret = avformat_alloc_output_context2(&ofmt_ctx, nullptr, nullptr, opt.output_path.c_str());
```

输入：

- 输出路径

输出：

- `ofmt_ctx`

为什么必须有：

- 输出容器的“壳子”必须先建好

### 第 6 步：创建 `StreamContext`

代码核心：

```cpp
StreamContext sc;
sc.type = type;
sc.input_index = input_index;
streams.push_back(sc);
in_to_ctx[input_index] = static_cast<int>(streams.size() - 1);
```

输入：

- 选中的输入流索引

输出：

- `streams` 里多了一个 `StreamContext`
- `in_to_ctx` 建好了映射

为什么必须有：

- 后面主循环读到输入包时，需要快速知道它属于哪个处理上下文

### 第 7 步：初始化每一路流

如果是 copy 模式：

- 创建输出流
- 拷贝 `codecpar`
- 建立 `in_to_out`

如果是 transcode 模式：

- 打开解码器
- 打开编码器
- 创建 `sws/swr/fifo`

### 第 8 步：打开解码器

函数：

- `open_decoder(ifmt_ctx, sc)`

它做了什么：

1. 根据输入流 `codecpar->codec_id` 找 decoder
2. 创建 `sc.dec_ctx`
3. 把输入流参数拷到 `dec_ctx`
4. 打开解码器

为什么必须有：

- 不打开解码器，就不能把压缩包变成原始帧

### 第 9 步：打开编码器

视频走：

- `open_video_encoder(...)`

音频走：

- `open_audio_encoder(...)`

它们做了什么：

- 创建 `sc.enc_ctx`
- 配置编码参数
- 创建输出流
- 把编码器参数写进输出流的 `codecpar`

视频编码器额外负责：

- 设置 `width/height`
- 设置 `pix_fmt = yuv420p`
- 设置 `time_base / framerate / gop / b_frames`
- 设置 `crf / preset`

音频编码器额外负责：

- 设置 `sample_rate / sample_fmt / ch_layout`
- 创建 `swr_ctx`
- 创建 `audio_fifo`

### 第 10 步：打开输出文件并写 header

代码：

- `avio_open(...)`
- `avformat_write_header(...)`

输入：

- 已经创建好的输出流信息

输出：

- 输出文件进入可写状态

为什么必须有：

- 没写 header，后面写包会失败

### 第 11 步：进入主循环

copy 模式走：

- `copy_mode_run(...)`

transcode 模式走：

- `transcode_mode_run(...)`

### 第 12 步：读输入 packet

代码：

```cpp
while ((ret = av_read_frame(ifmt_ctx, pkt)) >= 0) {
```

输入：

- 输入容器

输出：

- 一个 `AVPacket`

### 第 13 步：找到它属于哪一路流

在 transcode 模式里：

```cpp
auto it = in_to_ctx.find(pkt->stream_index);
StreamContext &sc = streams[it->second];
```

在 copy 模式里：

```cpp
auto it = in_to_out.find(pkt->stream_index);
```

为什么必须有：

- 先搞清楚当前包属于哪一路流，后面才知道怎么处理

### 第 14 步：送解码器，拿原始帧

代码：

```cpp
ret = avcodec_send_packet(sc.dec_ctx, pkt);
while ((ret = avcodec_receive_frame(sc.dec_ctx, frame)) >= 0) {
```

输入：

- `AVPacket`

输出：

- `AVFrame`

为什么必须有：

- 转码必须得到原始帧，才能后续处理和重编码

### 第 15 步：视频处理

函数：

- `encode_video_frame(sc, frame, ifmt_ctx, ofmt_ctx)`

内部步骤：

1. 分配目标视频帧
2. 用 `sws_getCachedContext()` 准备 `sws_ctx`
3. 用 `sws_scale()` 做像素格式/尺寸转换
4. 计算并换算 `frame->pts`
5. 把 frame 送编码器
6. 取出编码后的 packet 并写出

为什么必须有：

- 输入视频帧格式未必满足编码器要求

### 第 16 步：音频处理

音频分两步：

#### 第一步：写入 FIFO

函数：

- `fifo_write_audio(sc, frame)`

内部步骤：

1. `swr_convert()` 把输入音频转换成编码器要求的格式
2. 写入 `audio_fifo`

#### 第二步：从 FIFO 取样本并编码

函数：

- `encode_audio_from_fifo(sc, ofmt_ctx, false)`

内部步骤：

1. 从 FIFO 中读出足够样本
2. 组装新的音频 frame
3. 计算 `frame->pts`
4. 送编码器
5. 取出编码后的 packet 并写出

为什么要拆两步：

- 音频解码帧大小和编码器要求帧大小常常不一致

### 第 17 步：编码后写输出包

函数：

- `write_encoded_packets(sc, ofmt_ctx)`

它做了什么：

1. 从编码器取 `AVPacket`
2. `av_packet_rescale_ts()` 做时间基换算
3. 设置输出 `stream_index`
4. `av_interleaved_write_frame()` 写入容器

### 第 18 步：flush 解码器和编码器

这是初学者另一处经典难点。

为什么要 flush：

- 解码器可能还有缓存帧
- 编码器可能还有残留 packet
- 视频尤其可能因为 B 帧重排而没吐完

代码里分两段：

- 先 flush decoder：`avcodec_send_packet(sc.dec_ctx, nullptr)`
- 再 flush encoder：`avcodec_send_frame(sc.enc_ctx, nullptr)`

音频 flush 时还要把 FIFO 剩余样本编码完。

### 第 19 步：写 trailer 并释放资源

代码：

- `av_write_trailer(ofmt_ctx)`
- `cleanup_streams(streams)`
- `avformat_close_input(...)`
- `avio_closep(...)`
- `avformat_free_context(...)`

为什么必须有：

- trailer 写收尾信息
- 不释放资源会泄漏

## 第四部分：初学者最容易懵的地方

### 1. 为什么转码必须经历 decode -> encode？

因为转码本质上就是：

- 先把旧压缩格式解出来
- 再按新要求重新压进去

#### 为什么不能像 remux 那样直接改容器？

因为 remux 的前提是：

- 里面的编码格式本身不变

而转码意味着至少有一项变了：

- 编码格式
- 分辨率
- 像素格式
- 采样率
- 采样格式
- 声道布局

这些都要求程序先拿到原始帧，再重新编码。

#### 什么时候可以 copy？

只想换容器，不想改内容时。

#### 什么时候必须转码？

只要你想改媒体内容或编码参数，就必须转码。

### 2. 为什么视频还要做格式转换？

函数入口：

- `sws_scale()`

#### `sws_scale` 是干嘛的？

它负责：

- 像素格式转换
- 分辨率缩放

#### 为什么很多编码器喜欢 `yuv420p`？

因为：

- 兼容性最好
- 常见播放器/硬件解码器都支持
- H.264 常见输入格式就是它

#### 为什么输入帧不能直接丢给编码器？

因为输入解码出来的格式不一定是编码器支持的格式。  
比如输入可能是 `nv12`，但编码器要求 `yuv420p`。

### 3. 为什么音频还要做重采样？

函数入口：

- `swr_alloc_set_opts2()`
- `swr_convert()`

#### `SwrContext` 是干嘛的？

它负责：

- 采样率转换
- 采样格式转换
- 声道布局转换

#### 为什么音频要经过 `AVAudioFifo`？

因为音频编码器通常要求固定帧长，例如 AAC 常见是一帧固定若干样本。  
而解码器吐出来的音频帧，样本数未必刚好匹配编码器要求。

所以做法是：

- 先把转换后的样本写进 FIFO
- 凑够一帧再送编码器

### 4. 为什么要有 `StreamContext`？

因为 FFmpeg 自带对象只负责“容器”和“编解码器”层面的功能，  
但这份程序还需要一个“程序自己的一路流工作台”。

它解决的问题是：

- 让视频流和音频流各自维护自己的状态
- 不把大量变量散落在全局
- 主循环可以统一按 `StreamContext` 处理

它在主循环里的典型用法：

```cpp
auto it = in_to_ctx.find(pkt->stream_index);
StreamContext &sc = streams[it->second];
```

### 5. `in_to_ctx` 和 `in_to_out` 到底有什么区别？

#### `in_to_ctx`

- key：输入流索引
- value：`streams` 里 `StreamContext` 的下标

用途：

- 转码模式下，找到“当前输入包应该交给哪个 `StreamContext`”

#### `in_to_out`

- key：输入流索引
- value：输出流索引

用途：

- copy/remux 模式下，找到“当前输入包应该写到输出文件的哪一路流”

#### 为什么不能只保留一个 map？

因为这两个问题不是一个问题：

- `in_to_ctx` 解决“程序内部处理归谁管”
- `in_to_out` 解决“最终写到哪一路输出流”

### 6. 时间戳在这里怎么流转？

这是最核心也最容易糊的部分。

#### `time_base`

它表示“当前时间戳单位是什么刻度”。

比如：

- 输入流时间基可能是 `1/90000`
- 视频编码器时间基可能是 `1/25`
- 音频编码器时间基可能是 `1/48000`

所以同一个数字 `1000`，在不同 `time_base` 下代表的真实时间是不一样的。

#### 视频时间戳怎么处理？

在 `encode_video_frame()` 里：

```cpp
frame->pts = av_rescale_q(src_pts, in_stream->time_base, sc.enc_ctx->time_base);
```

意思是：

- 输入帧的 `pts` 原来属于输入流刻度
- 编码器要求自己的刻度
- 所以必须换算

#### 音频时间戳怎么处理？

在 `encode_audio_from_fifo()` 里：

```cpp
frame->pts = sc.next_audio_pts;
sc.next_audio_pts += frame->nb_samples;
```

因为音频编码器时间基是：

```cpp
AVRational{1, sample_rate}
```

所以音频时间轴可以按“采样点数”递增。

#### 为什么写包前还要 `av_packet_rescale_ts`？

因为编码器输出 packet 的时间基是：

- `enc_ctx->time_base`

而 muxer 写输出流时要求的是：

- `out_stream->time_base`

所以写包前必须再换一次刻度：

```cpp
av_packet_rescale_ts(pkt, sc.enc_ctx->time_base, out_stream->time_base);
```

如果这里做错了，常见后果包括：

- 音画不同步
- 时间戳非单调
- mux 失败
- 播放器进度条异常

## 第五部分：把视频链路和音频链路分开讲

### 视频链路

完整链路如下：

```text
读视频包
-> avcodec_send_packet(dec_ctx, pkt)
-> avcodec_receive_frame(dec_ctx, frame)
-> sws_scale 做像素格式/尺寸转换
-> 设置 frame->pts
-> avcodec_send_frame(enc_ctx, frame)
-> avcodec_receive_packet(enc_ctx, pkt)
-> av_packet_rescale_ts
-> av_interleaved_write_frame
```

视频链路的核心难点：

- 像素格式转换
- 视频 `pts` 换算
- 编码器 flush（尤其 B 帧）

### 音频链路

完整链路如下：

```text
读音频包
-> avcodec_send_packet(dec_ctx, pkt)
-> avcodec_receive_frame(dec_ctx, frame)
-> swr_convert 做采样转换
-> 写入 AVAudioFifo
-> 从 FIFO 取出足够样本
-> 组装新的音频 frame
-> 设置 frame->pts
-> avcodec_send_frame(enc_ctx, frame)
-> avcodec_receive_packet(enc_ctx, pkt)
-> av_packet_rescale_ts
-> av_interleaved_write_frame
```

音频链路的核心难点：

- 解码帧大小和编码器帧大小不一致
- 需要 FIFO 缓冲
- 音频时间轴按样本数递增

### 两条链路的相同点

- 都是 `packet -> frame -> packet`
- 都需要解码器和编码器
- 都需要时间戳处理
- 最后都要写到输出容器

### 两条链路的不同点

- 视频通常要 `sws_scale`
- 音频通常要 `swr_convert + audio_fifo`
- 视频的 `pts` 常来自输入帧换算
- 音频常按样本数推进 `pts`

## 第六部分：实现层面的理解

### 1. 为什么代码这样组织？

因为它把转码流程拆成了几个清晰层次：

- 参数层：`Options`
- 流状态层：`StreamContext`
- 初始化层：打开输入、创建输出、打开编解码器
- 主循环层：读包、处理、写包
- 资源管理层：cleanup / trailer

这是一种很适合教学、也很适合后续扩展的结构。

### 2. 如果没有 `StreamContext` 会怎样？

代码会很快变成这样：

- 视频一套变量
- 音频一套变量
- 到处写 `if (is_video)` / `if (is_audio)`
- 时间戳状态、输出流索引、转换器都散着放

后果就是：

- 难读
- 难扩展
- 难排查

### 3. 如果没有 `sws/swr/fifo` 会怎样？

#### 没有 `sws`

- 输入视频帧格式可能不被编码器接受
- 编码器直接报错或输出异常

#### 没有 `swr`

- 输入音频格式和编码器要求不一致
- 编码器无法正常编码

#### 没有 `fifo`

- 音频样本数对不齐
- 编码器可能拿不到它想要的固定帧大小
- 音频结尾容易出问题

### 4. 如果 flush 忘了做会怎样？

- 视频最后几帧丢失
- 音频尾巴丢失
- 编码器内部缓存没吐完
- 输出文件看起来“少了一点”

### 5. 如果时间戳处理错了会怎样？

常见现象：

- 音画不同步
- 播放器跳动异常
- `non-monotonic dts`
- 写包失败
- 播放卡顿或进度条异常

## 第七部分：最后的 3 份总结

### 总结 1：一句话说明转码在干什么

**转码就是把输入文件中的压缩音视频解出来，按新的编码和格式要求重新压缩，再写进新的输出容器。**

### 总结 2：这份代码的核心数据流

```text
输入文件
-> AVFormatContext(ifmt_ctx)
-> AVPacket
-> AVCodecContext(dec_ctx)
-> AVFrame
-> sws/swr/fifo
-> AVCodecContext(enc_ctx)
-> 新 AVPacket
-> AVFormatContext(ofmt_ctx)
-> 输出文件
```

### 总结 3：如果你自己从零写一个最小可用转码器，最少要写哪些步骤

最小步骤如下：

1. 解析输入输出参数
2. 打开输入文件
3. 探测流信息
4. 选择一路视频和一路音频
5. 创建输出上下文
6. 打开输入流解码器
7. 打开输出流编码器
8. 创建输出流并写 header
9. 主循环读取输入 packet
10. 解码得到 frame
11. 视频做 `sws_scale`，音频做 `swr_convert`
12. 把 frame 送编码器
13. 取出编码后的 packet
14. 换算时间戳
15. 写入输出文件
16. flush decoder / encoder
17. 写 trailer
18. 清理所有资源

## 一句话提醒

读这份代码时，请你始终抓住这条主线：

**packet 从哪里来，frame 在哪里产生，格式在哪里转换，packet 在哪里重新生成，最后又写到哪里去。**

只要这条主线清楚了，转码器的大框架你就真正入门了。
