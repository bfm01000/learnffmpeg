# Day 1：MP4 封装格式

## 1. MP4 文件结构概览

MP4（MPEG-4 Part 14）是一种**基于 box（atom）的容器格式**。整个文件由一系列嵌套的 box 组成，每个 box 有统一的头部结构，便于解析和扩展。

### 1.1 Box 的通用结构

每个 box 由以下部分组成（至少 8 字节）：

```
+--------+--------+--------+
| size   | type   | payload |
| 4 byte | 4 byte | ...     |
+--------+--------+--------+
```

- **size**（4 字节，大端）：整个 box 的字节数（含头部）。若为 1，表示使用 64 位扩展 size（大 box）。
- **type**（4 字节）：四个 ASCII 字符标识 box 类型，如 `ftyp`、`moov`、`trak`、`mdat`。
- **payload**：该 box 的数据内容，可能是子 box 或具体字段。

因此**定位任意 box**的步骤是：
1. 从文件或父 box 的起始位置读取 4 字节 → 得到 `size`。
2. 再读 4 字节 → 得到 `type`。
3. 若当前需要找的正是该 type，则此 box 的起始偏移、size、type 即已得到；否则跳到 `当前偏移 + size` 继续解析下一个 box。

## 2. 关键 Box 说明

### 2.1 `ftyp`（File Type Box）

- **作用**：声明文件类型、兼容品牌（brand）和版本。
- **位置**：通常出现在文件开头（在可选 `styp` 之后）。
- **与 moov/mdat 的关系**：无直接依赖，仅作类型标识，解析时往往先读 ftyp 再找 moov。

### 2.2 `moov`（Movie Box）

- **作用**：**整个 MP4 的“目录”**，包含所有与播放、索引、时间线相关的**元数据**，但不包含实际音视频采样数据。
- **内容**：内部是一组子 box，常见包括：
  - **mvhd**：整片的时间尺度、时长等。
  - **trak**：每条轨道一条（视频轨、音频轨、字幕轨等）。
  - **mvex**（可选）：若有 fragment 或某些封装会用到。
- **为何重要**：播放器必须先解析 `moov` 才能知道有多少轨、每轨的编码格式、时间戳、以及采样在 `mdat` 中的位置（通过 trak 内的 stbl 等）。若 `moov` 在文件末尾（如某些“web 优化”的 MP4），则需要先读到文件尾部才能开始解析。

### 2.3 `trak`（Track Box）

- **作用**：描述**一条媒体轨道**的元数据（如视频轨、音频轨）。
- **与 moov 的关系**：`trak` 是 `moov` 的子 box；一个 `moov` 里可以有多个 `trak`。
- **内容**：典型子结构包括：
  - **tkhd**：轨道 ID、时长、宽高（视频）等。
  - **mdia**：媒体信息，内含 **stbl**（Sample Table），其中 **stco/co64** 给出采样在文件中的偏移，**stsz** 给出采样大小，**stts** 等给出时间信息。解析 `trak` 才能把 `mdat` 里的数据与“第几秒、第几帧”对应起来。

简单记：**moov = 电影级元数据 + 多条 trak；trak = 单条轨道的元数据（含在 mdat 中的位置与时间）。**

### 2.4 `mdat`（Media Data Box）

- **作用**：**实际存储媒体数据**（音视频采样、字幕数据等）的 box。我们常说的“裸的 H.264/H.265/AAC 数据”就放在这里，但按 MP4 的索引（在 trak 的 stbl 里）被引用。
- **与 moov/trak 的关系**：
  - `moov` / `trak` 提供“目录和索引”；
  - `mdat` 提供“仓库里的货物”。
  - 播放时：根据 trak 中的 stco/co64、stsz、stts 等，在 mdat 中按偏移读取对应字节，再交给解码器。

### 2.5 三者关系小结

```
文件
├── ftyp          # 文件类型
├── moov          # 元数据总柜
│   ├── mvhd
│   ├── trak      # 轨道 1（如视频）
│   │   └── ... (tkhd, mdia, stbl...)
│   └── trak      # 轨道 2（如音频）
└── mdat          # 实际音视频数据（被 trak 中的索引指向）
```

- **moov**：元数据容器；**trak**：单轨元数据（含在 mdat 中的位置）；**mdat**：真实媒体数据。

## 3. 如何定位 moov、trak、mdat

1. **顺序扫描**：从文件开头（或 ftyp 之后）按“size + type”逐 box 扫描；若 type 为 `moov`、`trak`、`mdat` 则记录其起始偏移与 size。
2. **moov 在前 vs 在后**：若希望“边下边播”，常见做法是把 `moov` 放到文件末尾，此时需要从文件尾向前读，或先获取文件长度再在末尾附近查找 moov。
3. **trak**：只在 **moov** 内部递归子 box，遇到 type 为 `trak` 的即为一条轨道；可统计个数并解析每个 trak 的 tkhd/mdia 等获取更细信息。
4. **mdat**：与 moov 平级，在同一层级扫描到 type 为 `mdat` 的 box 即可；其 payload 为裸的媒体数据，具体如何切分由 trak 的 stbl 决定。

---

## 4. 实践建议（配合 Day 1 目标）

- 用十六进制编辑器或 `xxd` 打开一个 MP4，确认前 8 字节：前 4 字节为第一个 box 的 size（大端），后 4 字节为 type（如 `ftyp`）。
- 写一个小程序或脚本：只做“顶层 box 扫描”，依次打印每个顶层 box 的 type、size、在文件中的偏移；验证是否能看到 `moov` 和 `mdat`，以及 `moov` 内是否有多个 `trak`。
