# MP4 <-> H.264 面试速记

## 一页速记（先背这个）

### 1) 先定性：这是“转码”吗？
- MP4 -> H.264（前提视频轨本来是 H.264）：通常不是转码，是**解封装 + 码流重打包**。
- H.264 -> MP4：通常不是转码，是**重封装**。
- 只有当输入不是 H.264（如 HEVC）才需要解码再编码，才是“真转码”。

### 2) 一句话讲清两者差异
- H.264 是**编码标准/码流**（NALU 序列）。
- MP4 是**容器标准**（轨道、时间戳、索引、样本组织）。

### 3) MP4 -> H.264 的核心步骤
1. 按 `moov` 的采样表从 `mdat` 取视频样本（demux）。
2. 将 NALU 从 AVCC（长度前缀）转 Annex B（起始码）。
3. 从 `avcC` 取 SPS/PPS，按需要注入（常见关键帧前）。
4. 输出裸 `.h264`。

### 4) H.264 -> MP4 的核心步骤
1. 解析 Annex B NALU，识别访问单元（帧）。
2. 抽取 SPS/PPS，写入 `avcC`。
3. 生成时间轴和索引（`stts/ctts/stsz/stco/stss...`）。
4. NALU 转 AVCC 样本写入 `mdat`，元数据写入 `moov`。

### 5) SPS/PPS 补齐为什么重要
- 解码器在关键帧前必须拿到参数集，否则可能花屏/解码失败。
- 工程推荐：**每个 IDR 前注入 SPS/PPS**，提升随机切入和直播中途入会稳定性。

### 6) 常用命令（无损重封装）
```bash
# MP4 -> H.264
ffmpeg -i input.mp4 -an -c:v copy -bsf:v h264_mp4toannexb output.h264

# H.264 -> MP4（裸流需指定帧率）
ffmpeg -framerate 25 -i input.h264 -c:v copy output.mp4
```

### 7) 30 秒面试口播模板
“MP4 到 H.264 在视频本身已是 H.264 时，通常不是转码，而是解封装。先按 `moov` 索引从 `mdat` 拿样本，再把 AVCC 的长度前缀 NALU 转成 Annex B 起始码格式，同时把 `avcC` 中的 SPS/PPS 按需注入，最后输出裸码流。反向 H.264 到 MP4 则是重封装：解析 NALU、提取参数集写 `avcC`、生成时间戳和索引表后写入 `moov+mdat`。”

## 详细原理（面试深挖版）

### 1. 分层模型：编码层 vs 容器层
- **编码层（H.264）**：关心如何压缩图像，产物是 NALU（如 SPS/PPS/IDR/non-IDR/SEI）。
- **容器层（MP4）**：关心如何组织媒体、对齐多轨、时间戳、随机访问与元数据。

结论：同一段视频可以“编码不变、容器变化”。

### 2. MP4 内部怎么保存 H.264
- `mdat`：存样本字节。
- `moov`：存解释规则（轨道、时序、样本大小、偏移、关键帧表）。
- H.264 在 MP4 中通常是 **AVCC 格式**：
  - 样本中的 NALU 前是长度字段（1/2/4 字节）。
  - SPS/PPS 常在 `avcC`（extradata）中集中存放。

### 3. MP4 -> H.264 的“解封装 + 重打包”

#### Step A：样本提取（Demux）
- 读取 `moov` 中采样表定位 `mdat`：
  - `stco/co64`：样本偏移
  - `stsz`：样本大小
  - `stts/ctts`：时间信息
  - `stss`：关键帧索引

#### Step B：AVCC -> Annex B
- 输入：`[length][NALU][length][NALU]...`
- 输出：`[startcode][NALU][startcode][NALU]...`
- 本质是 NALU 边界表达方式转换，编码内容不变。

#### Step C：SPS/PPS 注入（补齐）
- 从 `avcC` 取参数集，转 Annex B NALU。
- 注入策略：
  - 仅文件头插入（简单，但随机切入弱）
  - 每个 IDR 前插入（更稳，推荐）

#### Step D：写出裸流
- 裸 `.h264` 不携带 MP4 那套索引/容器时序能力。

### 4. H.264 -> MP4 的“重封装”

#### Step A：解析 Annex B
- 通过起始码切 NALU，组装访问单元（一帧可能多个 slice）。

#### Step B：提取参数集
- 收集 SPS/PPS 写入 `avcC`，供解码器初始化。

#### Step C：构建时间与索引
- 裸流通常没有完整容器时间轴。
- muxer 依据输入帧率或时间信息生成 `PTS/DTS` 和采样表。
- 对有 B 帧场景，需正确处理显示顺序与解码顺序（`ctts`/`dts`）。

#### Step D：Annex B -> AVCC 后写 MP4
- 样本写进 `mdat`，结构信息写进 `moov`。

### 5. 为什么会“看起来是转换，实际不重编码”
- `-c copy` 表示直接拷贝编码数据，不做 decode/encode。
- 变化点在封装语义，不在压缩算法。

## SPS/PPS 补齐专题（面试高频）

### 1) 为什么缺了会出问题
- 解码前必须先拿到 SPS/PPS；否则关键帧也可能无法正确解码。

### 2) 从哪里补
- MP4 的 `avcC`（extradata）里。

### 3) 补到哪里
- 常见为 IDR 前：`[SPS][PPS][IDR]`（可带 AUD/SEI）。

### 4) 推荐策略
- 存储文件可考虑文件头+关键点插入。
- 直播/低延迟场景优先每个 IDR 前注入，增强中途入会能力。

## 面试优化建议（直接加分）

### 1) 回答结构优化（先结论后细节）
- 先给一句定义：是否转码。
- 再讲分层：编码 vs 容器。
- 再讲流程：输入结构 -> 转换步骤 -> 输出结构。
- 最后讲工程坑：SPS/PPS、B 帧时序、帧率设定、seek 稳定性。

### 2) 常见追问与高分回答要点
- **Q: 为什么 H.264->MP4 要指定 framerate？**
  - A: 裸流缺容器级时间信息，需给 muxer 时基来源，否则时长/播放速度可能异常。

- **Q: 为什么提取后 `.h264` 不能像 MP4 那样好 seek？**
  - A: 裸流没有 MP4 的采样索引和时间表，随机定位能力弱，通常需扫描关键帧。

- **Q: 什么情况下必须转码？**
  - A: 编码不一致（如 HEVC->H.264）、分辨率/码率/帧率策略调整、像素格式/兼容性要求变化。

### 3) 实操建议（避免翻车）
- 先 `ffprobe` 确认 codec，再决定能否 `-c copy`。
- 裸流封装时明确输入帧率。
- 对直播或中途入会场景，确认 IDR 前参数集注入策略。
- 做回归验证：可播放性、关键帧切入、音视频同步（多轨时）。

工程上关键是参数集注入和时序正确性，尤其 B 帧下的 PTS/DTS 关系，以及裸流封装时帧率设置，否则会出现解码失败或播放速度异常。”
