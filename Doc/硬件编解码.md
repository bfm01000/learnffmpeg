# 硬件解码与 AVFrame 理解指南

这是一份**纯原理版**说明文档，不依赖任何具体项目实现。  
目标是帮助已经理解**软件解码**的同学，建立对**硬件解码**和 **AVFrame** 的正确认知。

---

## 1. 先建立总认知

很多初学者会把硬件解码理解成“另一套完全不同的流程”。  
其实不是。

无论软解还是硬解，主流程骨架都类似：

1. 创建解码器
2. 打开解码器
3. 输入压缩码流
4. 输出解码结果
5. 交给后续模块使用

真正的差异主要在 4 个地方：

1. **谁负责解码**
   - 软解：CPU
   - 硬解：平台专用视频硬件

2. **输出结果放在哪里**
   - 软解：通常在 CPU 内存
   - 硬解：通常在 GPU 或平台硬件缓冲区

3. **后续如何使用输出帧**
   - 软解：可以直接按像素平面读取
   - 硬解：通常先判断是否保留在 GPU，还是转回 CPU

4. **兼容性和降级策略**
   - 软解：相对稳定
   - 硬解：更依赖平台、驱动、设备能力

## 2. 软件解码和硬件解码的本质区别

## 软件解码

软件解码时，压缩码流的解析和还原由 CPU 完成。  
输出结果通常是普通像素帧，例如：

- `YUV420P`
- `NV12`
- `P010`

这类帧一般可以直接这样理解：

- `data[0]`、`data[1]`、`data[2]` 指向像素平面
- `linesize[]` 表示每行跨度
- 后续算法或渲染模块可以直接读取这些数据

一句话：

> 软件解码输出的 `AVFrame`，通常就是“像素数据本体”。

## 硬件解码

这一节只写一件事：

> **硬解时 `AVFrame` 和软解时 `AVFrame` 到底有什么区别，应该怎么用。**

## 先看软解和硬解的直接区别

| 维度 | 软件解码输出的 `AVFrame` | 硬件解码输出的 `AVFrame` |
|---|---|---|
| `frame->format` | 常见是普通像素格式，如 `YUV420P`、`NV12`、`P010` | 常见是硬件格式，如 `AV_PIX_FMT_VIDEOTOOLBOX`、`AV_PIX_FMT_D3D11`、`AV_PIX_FMT_CUDA` 等 |
| `data[]` 的含义 | 通常就是像素平面地址 | 往往不是普通像素地址，可能是平台对象、纹理、surface、设备内存指针 |
| `linesize[]` 的含义 | 普通图像平面步长 | 可能仍有意义，但语义依赖硬件类型，不能默认按软件帧理解 |
| `hw_frames_ctx` | 通常为空，或者不重要 | 往往非常关键，描述硬件上下文、设备类型、底层软件格式 |
| 使用方式 | 直接读像素平面 | 先识别硬件类型，再提取底层资源，决定走 GPU 还是先转软件帧 |

一句话说清楚：

- **软解 `AVFrame`：重点是像素数据**
- **硬解 `AVFrame`：重点是硬件资源引用**

## 软解时，`AVFrame` 怎么用

软解时，`AVFrame` 通常可以直接按“图像数据”理解：

- `data[0]` 是 Y 平面
- `data[1]` 是 U 或 UV 平面
- `data[2]` 是 V 平面
- `linesize[]` 是对应步长

典型写法：

```cpp
AVFrame* frame = receive_frame(decoder);

uint8_t* y = frame->data[0];
uint8_t* uv = frame->data[1];
int yStride = frame->linesize[0];

process_yuv(y, uv, yStride, frame->width, frame->height);
```

这里你操作的就是“像素本身”。

## 硬解时，`AVFrame` 怎么看

硬解时，最容易犯的错误是：

> 还按软解思路，默认 `data[0]`、`data[1]`、`data[2]` 一定就是 YUV 平面。

这往往不对。

硬解时你应该先问：

1. `frame->format` 是什么
2. 这个平台的硬件帧把资源放在哪个字段
3. 后面是继续走 GPU，还是转回 CPU

也就是说，硬解时真正的使用入口通常不是“直接读像素”，而是“先把底层硬件资源取出来”。

## 各平台硬解时，`AVFrame` 里到底装了什么

| 平台 | `frame->format` | `AVFrame` 里真正装的东西 | GPU-only 渲染怎么用 |
|---|---|---|---|
| Apple | `AV_PIX_FMT_VIDEOTOOLBOX` | `data[3]` 是 `CVPixelBufferRef` | 用 `CVMetalTextureCache` 或 `CVOpenGLESTextureCache` 生成纹理，直接渲染 |
| Android | `AV_PIX_FMT_AIMAGEREADER_HWBUFFER` / `AV_PIX_FMT_AHARDWAREBUFFER` | `data[3]` 是 `AHardwareBuffer*` | 导入成 `EGLImage` / GL 纹理 / Vulkan image，直接渲染 |
| Android 旧路径 | `AV_PIX_FMT_MEDIACODEC` | MediaCodec surface 相关对象 | 更老的 surface 路线，通常不作为主渲染路径依赖 |
| Windows D3D11 | `AV_PIX_FMT_D3D11` | `data[0]` 是 `ID3D11Texture2D*`，`data[1]` 是 subresource/index | 直接交给 D3D 渲染，或做 D3D11 共享纹理 |
| Windows DXVA2 | `AV_PIX_FMT_DXVA2_VLD` | DXVA surface | 更老的路径，通常先转/拷到 D3D11 或 Host 再继续处理 |
| Windows/Linux CUDA | `AV_PIX_FMT_CUDA` | `data[0]` / `data[1]` 是 CUDA device memory 平面指针，`hw_frames_ctx` 里有 CUDA context 和 `sw_format` | 用 CUDA-OpenGL / CUDA-Vulkan 互操作，或 device-to-device 拷贝到 GPU 纹理 |
| Windows Intel QSV | `AV_PIX_FMT_QSV` | QSV surface，核心信息通常在 `hw_frames_ctx` | 一般不直接手读，更常见是桥接到 D3D11 再渲染 |

## 按平台看，怎么从 `AVFrame` 里取资源

### Apple

如果：

```cpp
frame->format == AV_PIX_FMT_VIDEOTOOLBOX
```

那通常表示：

- `data[3]` 里是 `CVPixelBufferRef`

GPU-only 时，不是去读 YUV 平面，而是：

```cpp
CVPixelBufferRef pixelBuffer = reinterpret_cast<CVPixelBufferRef>(frame->data[3]);
// 用 CVMetalTextureCache 或 CVOpenGLESTextureCache 生成纹理
```

如果后面是 CPU 算法，才需要把 `CVPixelBufferRef` 转成普通软件帧。

### Android

如果：

```cpp
frame->format == AV_PIX_FMT_AIMAGEREADER_HWBUFFER
// 或 AV_PIX_FMT_AHARDWAREBUFFER
```

那通常表示：

- `data[3]` 里是 `AHardwareBuffer*`

GPU-only 时更自然的用法是：

```cpp
AHardwareBuffer* hardwareBuffer = reinterpret_cast<AHardwareBuffer*>(frame->data[3]);
// 导入成 EGLImage / GL texture / Vulkan image
```

这里要特别记住一点：

- Android 硬件缓冲区经常和底层输出池绑定
- 如果 `AVFrame` 提前释放，内容可能被后续帧复用

所以很多时候要么保持 `AVFrame` 活着，要么额外持有 `AHardwareBuffer` 引用。

### Windows D3D11

如果：

```cpp
frame->format == AV_PIX_FMT_D3D11
```

那通常表示：

- `data[0]` 是 `ID3D11Texture2D*`
- `data[1]` 是对应 subresource/index

这时 `AVFrame` 的重点不是像素平面，而是这块 D3D 纹理：

```cpp
ID3D11Texture2D* tex = reinterpret_cast<ID3D11Texture2D*>(frame->data[0]);
int index = reinterpret_cast<intptr_t>(frame->data[1]);
// 直接交给 D3D renderer
```

### Windows DXVA2

如果：

```cpp
frame->format == AV_PIX_FMT_DXVA2_VLD
```

那它表示的是 DXVA surface。  
它仍然是硬件帧，但很多实际链路里不会直接把它当最终渲染输入，而是：

1. 先转成 D3D11 资源
2. 或先拷到 Host
3. 再继续处理

### Windows/Linux CUDA

如果：

```cpp
frame->format == AV_PIX_FMT_CUDA
```

那通常表示：

- `data[0]` / `data[1]` 是设备内存平面指针
- `linesize[]` 是设备侧 pitch
- `hw_frames_ctx` 保存 CUDA 上下文和底层 `sw_format`

这时不能把 `data[0]` 当普通 CPU 内存指针。  
更合理的理解是：

```cpp
// data[0]/data[1] 是 device memory
// hw_frames_ctx 里有 CUDA context
// 后续做 CUDA-OpenGL / CUDA-Vulkan 互操作
```

如果后面是 CPU 算法，就需要先做硬件帧到软件帧的 transfer。

### Intel QSV

如果：

```cpp
frame->format == AV_PIX_FMT_QSV
```

那通常表示：

- 它是一块 QSV surface
- 核心信息更多放在 `hw_frames_ctx`

这类帧一般不建议按“普通 `data[]` 平面”去理解。  
更常见做法是先桥接到图形链路，再渲染。

## 三种真实场景下，硬解 `AVFrame` 怎么用

### 场景 1：播放器预览

目标是直接显示视频。

正确思路：

1. 拿到硬件帧
2. 从 `AVFrame` 中取出平台资源
3. 直接交给 GPU 渲染

这时不要先转软件帧。

### 场景 2：GPU 特效

如果后面还是 GPU 处理，例如：

- 美颜
- LUT
- shader 滤镜

正确思路也是：

1. 保持硬件帧形态
2. 提取底层资源
3. 继续在 GPU 上处理

### 场景 3：CPU 算法

如果后面是：

- OpenCV
- 逐像素处理
- CPU 模型前处理

那正确思路是：

1. 先把硬件帧转成软件帧
2. 再按软解方式访问 `data[]`

## 最后一句最关键

> 软解时，你主要关心的是 `data[]` 里有什么像素。  
> 硬解时，你主要关心的是 `AVFrame` 里引用的是哪块硬件资源，以及这块资源该怎么交给 GPU 或怎么转回 CPU。

## 3. 为什么硬件解码通常更快

视频解码的核心过程包括：

- 熵解码
- 反量化
- 反变换
- 运动补偿
- 环路滤波

这些操作计算量很大。  
硬件解码的优势在于：平台上有专门的视频解码单元，专门为这些任务优化。

因此相比软件解码，硬解通常有这些好处：

- CPU 占用更低
- 功耗更低
- 长时间播放更稳定
- 高分辨率和高码率场景更容易撑住

但要注意：

> 硬解“更快”不一定等于整条链路一定更快。

如果解码后立刻又把数据从 GPU 拷回 CPU，再做大量 CPU 处理，性能优势可能会被抵消。

## 4. 硬解的完整心智模型

理解硬解，建议把整条链路拆成下面几步：

1. 输入压缩码流
2. 平台硬件解码器解码
3. 产生硬件帧
4. 决定后续走向

硬件帧之后通常有两种路线：

### 路线 A：继续留在 GPU

适合：

- 播放
- 预览
- GPU shader 滤镜
- GPU 渲染链路

优点：

- 少拷贝
- 性能好
- 更能发挥硬解优势

### 路线 B：转回 CPU

适合：

- CPU 图像算法
- OpenCV
- 逐像素处理
- 只接受普通内存帧的模块

缺点：

- 有额外拷贝成本
- 可能削弱硬解收益

## 5. `AVFrame` 在软解和硬解中的不同角色

## 软解里的 `AVFrame`

软解时，`AVFrame` 通常表示：

- 图像宽高
- 像素格式
- 各平面地址
- 各平面的 stride

你可以把它理解成：

> “一张可以直接访问像素内容的图像帧”

## 硬解里的 `AVFrame`

硬解时，`AVFrame` 更多表示：

- 当前帧属于哪种硬件格式
- 当前帧关联哪个硬件设备
- 当前帧引用了哪块平台资源
- 如果需要转软件帧，该怎么转

你可以把它理解成：

> “一张硬件帧的说明书或包装壳”

所以初学者最容易犯的错误就是：

> 看到 `AVFrame` 就默认想直接读 `data[0]`。

这在硬解里往往是不对的。

---

## 6. 读硬解 `AVFrame` 时最重要的 3 件事

拿到一个硬解输出帧时，先做这三步判断：

### 第一步：看像素格式

像素格式会告诉你：

- 这是普通软件帧
- 还是某个平台的硬件帧

如果是普通格式，通常可以直接读像素平面。  
如果是硬件格式，就要进入第二步。

### 第二步：看资源类型

要搞清楚这个 `AVFrame` 里装的是：

- 平台图像缓冲区
- GPU 纹理
- GPU surface
- 设备侧平面指针
- 或者某种平台专用对象

### 第三步：决定是走 GPU 还是回 CPU

- 如果只做 GPU 渲染：尽量不要回 CPU
- 如果后续是 CPU 算法：再转普通软件帧

这三个问题，比“函数调用顺序”更重要。

## 7. 各平台硬解帧的大体形态

这一节专门回答一个很实际的问题：

> 如果是硬件解码，应该怎么使用 `AVFrame`？

最重要的不是背定义，而是先搞清楚下面这 4 件事：

1. 当前平台是什么
2. `frame->format` 是什么
3. `AVFrame` 里真正装的是什么对象
4. 如果只做 GPU 渲染，应该怎么把它交给渲染链路

先看总表。

## 7.2 Apple: `VideoToolbox`

这类最典型。

核心特征：

- `frame->format == AV_PIX_FMT_VIDEOTOOLBOX`
- `frame->data[3]` 里通常放的是 `CVPixelBufferRef`

### GPU-only 怎么用

不要先转成普通 YUV 帧。  
而是：

1. 从 `data[3]` 取出 `CVPixelBufferRef`
2. 用系统纹理缓存把它变成 GPU 纹理
3. 让 shader 直接采样

常见方式：

- Metal: `CVMetalTextureCache`
- OpenGL: `CVOpenGLESTextureCache`

### 如果要给 CPU 算法

那才需要做：

1. 从 `data[3]` 取出 `CVPixelBufferRef`
2. 拷贝或转换成普通软件帧
3. 再访问 `data[]`

### 怎么理解

Apple 平台的“读 `AVFrame`”不是先读像素，而是：

> 先把 `AVFrame` 当成 `CVPixelBufferRef` 的容器。

下面这个例子体现的就是“如果要回 CPU，就先从 `data[3]` 取出 `CVPixelBufferRef` 再转换”：

```cpp
if (frame->format == AV_PIX_FMT_VIDEOTOOLBOX) {
    CVPixelBufferRef pixelBuffer = reinterpret_cast<CVPixelBufferRef>(frame->data[3]);
    // 如果后续是 CPU 算法，再把 pixelBuffer 转成普通软件帧
}
```

---

## 7.3 Android: `AHardwareBuffer`

Android 上真正适合 GPU-only 渲染的硬解输出，通常是：

- `AV_PIX_FMT_AIMAGEREADER_HWBUFFER`
- `AV_PIX_FMT_AHARDWAREBUFFER`

这条路线和 Apple 有点像，也是从 `AVFrame` 里取平台对象，但对象不是 `CVPixelBufferRef`，而是 `AHardwareBuffer*`。

### GPU-only 怎么用

更自然的路径是：

1. 从 `frame->data[3]` 取出 `AHardwareBuffer*`
2. 把这个 buffer 导入图形系统
3. 变成 GL/Vulkan 能消费的图像资源
4. 直接渲染

常见方式：

- `AHardwareBuffer -> EGLImage -> GL texture`
- `AHardwareBuffer -> Vulkan image`

### 代码层面怎么理解

如果你在代码里看到类似这种判断：

```cpp
if (frame->format == AV_PIX_FMT_AIMAGEREADER_HWBUFFER) {
    AHardwareBuffer* hardwareBuffer = (AHardwareBuffer*)frame->data[3];
    // 后续把 hardwareBuffer 导入 GPU
}
```

它表达的意思就是：

> 这张 `AVFrame` 不是普通 YUV 帧，而是在 `data[3]` 里带着一个 Android 硬件缓冲区。

### Android 特别要注意的坑

Android 这类硬件缓冲区通常和底层解码输出池、`AImageReader` 之类的对象关联更紧。

这意味着：

- `AVFrame` 释放后，底层 image 可能被复用
- 同一个 `AHardwareBuffer` 的内容，后续可能被新帧覆盖

所以 GPU 渲染时经常要注意两件事：

1. 保持 `AVFrame` 生命周期足够长
2. 或者显式持有一份 `AHardwareBuffer` 引用

### 如果不是这条路线

Android 还有更老的 `AV_PIX_FMT_MEDIACODEC` surface 路线。  
它本质上是另一种平台对接方式，但现在很多工程里已经不是首选主路径。

心智模型：

> Android 硬解时，你更应该先想“这是不是一个 `AHardwareBuffer*`”，而不是先想 YUV 平面。

---

## 7.4 Windows: `D3D11`

这条路线最适合 D3D 渲染链路直接消费。

核心特征通常是：

- `frame->format == AV_PIX_FMT_D3D11`
- `data[0]` 是 `ID3D11Texture2D*`
- `data[1]` 是 subresource 或 index
- `hw_frames_ctx` 描述它属于哪个硬件帧上下文

### GPU-only 怎么用

最顺手的方式通常是：

1. 判断 `frame->format == AV_PIX_FMT_D3D11`
2. 从 `data[0]` 取出 `ID3D11Texture2D*`
3. 用 `data[1]` 作为对应子资源索引
4. 直接交给 D3D11 renderer，或做共享纹理

### 如果要给 CPU

一般需要：

1. 把显卡资源拷贝到 CPU 可访问的 staging 资源
2. 再转成普通软件帧

### 怎么理解

这里的 `AVFrame` 更像是在说：

> “这张图在一块 D3D11 texture 上，你拿这个 texture 去渲染。”

而不是：

> “这张图就在 `data[0]` 指向的一段普通 YUV 内存里。”

---

## 7.5 Windows: `DXVA2`

这是更老一点的 Windows 硬解路线。

核心特征：

- `frame->format == AV_PIX_FMT_DXVA2_VLD`
- 底层通常对应 DXVA surface

### GPU-only 怎么用

理论上可以继续走 Windows 图形链路，但在很多工程里更常见的做法是：

1. 先把 DXVA2 surface 转成更容易处理的 D3D11 资源
2. 或者先拷到 Host
3. 再进入后续链路

### 怎么理解

这类路径依然属于“硬件帧”，只是相比 D3D11 更老、兼容层更多。

心智模型：

> 它依然不是普通软件帧，只是工程里很多时候不会直接把它作为最终渲染输入。

---

## 7.6 Windows/Linux: `CUDA`

这类路径和 Apple / Android 最大的不同是：

> 它不一定把“核心对象”放在 `data[3]`。

更常见的理解方式是：

- `frame->format == AV_PIX_FMT_CUDA`
- `hw_frames_ctx` 里有 CUDA 设备上下文
- `data[0]` / `data[1]` 指向 device memory 的平面
- `sw_format` 说明底层实际是 `NV12`、`P010` 等什么软件格式

### GPU-only 怎么用

常见思路：

1. 从 `hw_frames_ctx` 拿到 CUDA context
2. 把目标纹理注册成 CUDA graphics resource
3. 使用 device-to-device 拷贝
4. 后续 shader 再采样纹理

常见形态：

- CUDA-OpenGL 互操作
- CUDA-Vulkan 互操作
- 直接拷到 GPU 纹理或 GPU image

### 代码使用上应该怎么想

这里不要先把 `data[0]` 当普通 CPU 地址。  
更合理的理解是：

- `data[0]` / `data[1]` 是 GPU 设备内存地址
- `linesize[]` 也是设备侧 pitch
- 必须在正确的 CUDA 上下文里使用

### 如果要给 CPU

那就需要显式做一次：

1. 硬件帧到软件帧的 transfer
2. 再把结果当普通 `AVFrame` 处理

心智模型：

> CUDA 硬解帧不是“普通内存中的图像”，而更像“设备内存平面 + 上下文”。

---

## 7.7 Windows Intel: `QSV`

QSV 这类帧对初学者最容易产生误解，因为你可能看不到一个像 Apple 的 `data[3]`，也不像 D3D11 那样能简单把 `data[0]` 看成纹理对象。

更稳妥的理解方式是：

- `frame->format == AV_PIX_FMT_QSV`
- 它通常代表一块 QSV surface
- 核心信息更多放在 `hw_frames_ctx`
- 更适合通过硬件帧机制继续流转

### GPU-only 怎么用

很多实际工程里不会直接“手读” QSV frame。  
更常见的是：

1. 桥接到 D3D11
2. 再交给 D3D 渲染链路

这样往往比直接把 QSV 当成某种普通平面数据来理解更自然。

心智模型：

> QSV 更像“平台内部视频 surface”，而不是“我直接读 `data[]` 就能处理的图像”。

---

## 8. 为什么硬解时不能默认直接读 `data[0]`

这是最值得单独强调的一点。

### 软解时

很多情况下你可以这样想：

- `data[0]` 是 Y
- `data[1]` 是 U 或 UV
- `data[2]` 是 V

### 硬解时

`data[]` 的含义可能完全不同：

- 可能是平台对象地址
- 可能是 GPU 纹理句柄
- 可能是设备侧平面指针
- 可能只是某个描述结构的一部分

所以正确思路不是：

> “先去读 `data[0]` 里面的像素”

而是：

> “先判断这个 frame 属于哪种硬件资源，再决定怎么取用”

---

## 9. 只做 GPU 渲染时，应该怎么思考

如果你的目标是“解码后直接渲染”，那么最理想的思路就是：

1. 解码得到硬件帧
2. 尽量不做 CPU 拷贝
3. 尽量不转普通软件帧
4. 直接把硬件帧桥接为 GPU 纹理或图像对象
5. 让 shader 去采样

这种方式常被称为：

- 零拷贝
- 近零拷贝
- GPU-only 路线

它的核心优势是：

- 减少数据搬运
- 降低 CPU 压力
- 缩短处理路径

但它也更依赖：

- 平台能力
- 驱动质量
- 图形 API 互操作能力
- 资源生命周期管理

---

## 10. 如果要把硬解帧交给 CPU 算法

不是所有场景都适合 GPU-only。

如果后续模块需要：

- CPU 图像处理
- OpenCV
- 软件滤镜
- 导出为普通图像

那么通常需要做一次“硬件帧 -> 软件帧”的转换。

常见方式有两类：

1. 平台专用转换
   - 通过平台 API 拷出一份普通图像数据

2. 通用硬件帧传输
   - 通过媒体框架提供的硬件帧传输接口转成软件帧

这一步之后，`AVFrame` 才重新变回你熟悉的样子：

- `data[]` 是像素平面
- `linesize[]` 是步长
- 可以像软解一样访问

---

## 11. 初学者最容易混淆的三个点

## 混淆点 1：硬解输出到内存，就等于软解

不一定。

有些场景里虽然最后拿到的是普通内存帧，但解码过程本身仍然是硬件解码。  
只是硬件解码之后，结果被额外 copy 或转换到了 CPU 内存。

所以要区分：

1. **纯软解**
2. **硬解后继续保留硬件帧**
3. **硬解后再转成软件帧**

这三种不是一回事。

## 混淆点 2：用了 `AVFrame` 就说明是软件帧

不对。

无论软解还是硬解，很多媒体框架最终都会把结果包在 `AVFrame` 中。  
关键不是“有没有 `AVFrame`”，而是：

- `AVFrame` 的像素格式是什么
- 它关联的是普通内存还是硬件资源

## 混淆点 3：硬解一定比软解快

也不绝对。

如果你：

1. 硬解
2. 立刻拷回 CPU
3. 再做重 CPU 处理

那么整条链路不一定比软解更有优势。

真正高性能的关键是：

> 解码之后能不能继续在 GPU 侧消费。

---

## 12. 给初学者的一套判断口诀

以后你拿到一个解码输出帧，建议按这套顺序思考：

### 1. 先问：这是软件帧还是硬件帧

先看像素格式和上下文信息。

### 2. 再问：这个 frame 里面装的是像素，还是资源

- 软件帧：通常是像素
- 硬件帧：通常是资源

### 3. 再问：后续要走 CPU 还是 GPU

- 走 GPU：尽量保留硬件帧
- 走 CPU：再做转换

### 4. 最后再决定怎么读

- 软件帧：按平面读
- 硬件帧：按平台资源语义读

## 13. 一张总表

| 场景 | `AVFrame` 更像什么 | 典型使用方式 |
|---|---|---|
| 软件解码 | 普通像素帧 | 直接访问像素平面 |
| Apple 硬解 | 平台图像缓冲区 | 直接桥接到 GPU 纹理 |
| Android 硬解 | 平台硬件缓冲区 | 导入 GL/Vulkan 后渲染 |
| Windows 硬解 | 显卡资源或视频 surface | 直接交给图形链路 |
| NVIDIA/GPU 硬解 | 设备内存平面 + 设备上下文 | GPU 内部拷贝或直接消费 |
| 硬解转软件帧 | 普通像素帧 | 再像软解一样处理 |

---

## 14. 最后的总结

如果只记住一句话，请记住这句：

> 软件解码时，`AVFrame` 通常装的是“像素数据”；硬件解码时，`AVFrame` 通常装的是“硬件资源的引用或描述”。

所以理解硬解最关键的不是死记函数调用顺序，而是搞清楚：

1. 当前输出帧是软件帧还是硬件帧
2. 这张帧的数据到底在 CPU 还是 GPU
3. 后续链路是想继续留在 GPU，还是转回 CPU

把这三件事想清楚，硬件解码的理解就会一下子顺很多。
